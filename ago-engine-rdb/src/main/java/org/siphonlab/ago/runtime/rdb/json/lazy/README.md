JSON 半结构化 + lazy 模式

本模式为完全实现的 ObjectRefInstance 模式.

slots 按需完全加载. 采用 json/jsonb 存储 slots.

本模式特别适合 workflow.

技术上存在各种排列组合, 但根据应用场景 workflow 就应当选用该模式, 至于是否在数据侧进行运算, 连 slots 也完全是虚拟的, 走 reactive 模式, 
这个要看实际情况, 如果数据库是我自己写的, 它就是一个数据库自带的开发语言, 高度集成, 我觉得可以投入. 
但要适配各种数据库, 很不明智, 有的数据库甚至 JSON 就是一个字符串, 而即使向 PG 支持的这么好的, 它也是一个整体存储.

如果不做在数据库内，它实际上违背基本的分布式原则，读写数据库频繁，会对数据库造成很大压力，毕竟网络不是内存总线。

此外，尽管关系型数据库也能搞 workflow, 该模式也跑通了, 但以跑一次流程得到一个实体来论, 还是 mongodb 更合适.

mongodb 的文档的更新是部分更新, 不整体重写, 此外, 如果将文档理解为一个层次型数据库, 将 class 等一一放入,
则正好一个 application 可以当作一个文档.

我认为这种模式更能满足 workflow 的需求.

当然, 从 RDB 来说, 这几张固定表和 activiti 建的表也很相似, 也值得继续做完.
